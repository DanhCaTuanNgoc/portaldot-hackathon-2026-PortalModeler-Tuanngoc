2. Product Summary

PortalModeler Proof is a visual Portaldot workbench that turns canvas nodes into real local blockchain actions. The stable MVP flow connects to a local node, checks balance, estimates POT fees, submits a real Transfer POT transaction, and shows the resulting extrinsic, block, and events.

The key idea is a safe runner inside the Vite app: visual nodes can only execute whitelisted repo scripts or labeled local helpers, not arbitrary shell commands. The repo also includes an ink! Membership contract workflow as extended functionality, while the checkpoint-critical demo uses the stable Transfer POT flow.

3. Current MVP Status

What already works:
1. The workbench connects to a local Portaldot node at ws://127.0.0.1:9944, checks local health, and shows whether RPC, artifacts, and contract state are ready.
2. The visual board supports real workflow execution: users can run one node, run from a selected node, or run the full graph in order, with node status updates and run logs.
3. The stable Transfer POT flow queries Alice's balance, estimates the transaction fee through payment_queryInfo, submits a real Balances.transfer_keep_alive transaction, and displays the extrinsic hash, block hash, and chain events.
4. The Vite safe runner maps each executable node to a whitelisted script, which keeps the demo reproducible and prevents arbitrary command execution from the browser.
5. The AI Flow Builder can turn a natural-language prompt into a safe visual Transfer POT workflow by creating only approved node types and connections.
6. The Membership contract, deploy/call scripts, artifact checks, and state readers are implemented as extended contract functionality, but they are not the checkpoint-critical proof flow.

What is unfinished:
1. The planner is still limited in scope and is not a fully general workflow engine.
2. Some advanced workflow management features are still presentation-oriented rather than fully productized.

What is mocked (must declare):
1. Check Account in the Vite runner is a local helper that returns signer information rather than a full chain query path.
2. watchEvents, decodeEvents, exportWorkflow, exportCommands, saveWorkflow, loadWorkflow, and generateReport are browser-generated helpers, not on-chain actions.
3. AI Flow Builder is a local deterministic planner stub. No model API is connected yet; it only converts a prompt into whitelisted visual nodes and does not mock blockchain execution.

5. POT Gas / Fee Status

Can you show POT gas / fee usage? Yes

Which action consumes POT?
- The checkpoint-critical action is Transfer POT. It uses the Balances.transfer_keep_alive extrinsic, which moves the configured amount of POT from Alice to the recipient and also charges Alice a transaction fee.
- The fee is estimated before submission with payment_queryInfo. In the current local dry-run, transferring 1000000000000 base units (0.010000 POT) estimates a fee of 270205654 base units (0.000003 POT).
- The extended Membership join action is also payable at the contract level: calling join requires sending the configured join fee in base units. This exists in the contract workflow, but the stable demo uses Transfer POT as the primary onchain proof.

How will you show it?
- Use scripts/transfer.py with --dry-run-only to display the estimated fee before submitting.
- If needed, run the transfer for real and show the extrinsic hash, block hash, Balances.Transfer event, and System.ExtrinsicSuccess.

Evidence:
- scripts/transfer.py prints the estimated fee using payment_queryInfo and can also submit a real local transfer.
- The current local dry-run output shows an estimated fee of 270205654 base units, which is 0.000003 POT.

6. One Core Demo Flow

1. Start or verify the local Portaldot node at ws://127.0.0.1:9944.
2. Open the PortalModeler Workbench and refresh local health.
3. Run Check Balance to display Alice's POT balance.
4. Run Transfer POT with 0.010000 POT.
5. Show the run logs with the fee estimate, extrinsic hash, block hash, and events.

What could fail in this demo?
1. The local RPC node may be offline, unreachable, or pointed to the wrong endpoint.
2. The local chain state may differ after a reset, which can make addresses stale or affect transaction results.
